# SFT 数据生成工作流

基于 LangGraph 构建的 SFT（监督微调）数据生成工作流，用于批量生成内外规条款匹配 QA 对，训练冲突判别大模型。

## 项目概述

### 目标

- 生成 **10,000-20,000 条训练数据** 和 **≥500 条验证数据**
- 用于微调冲突判别大模型
- 输出标准 SFT 训练格式（LLaMA Factory Alpaca 格式）

### 核心特性

1. **并发处理** - 使用 ThreadPoolExecutor 实现 20 并发 LLM 调用
2. **分层筛选** - 基于匹配特征分层采样，优化标签分布
3. **主动构造冲突** - 对匹配 pair 生成冲突场景 QA，增加冲突样本
4. **语义要素预筛选** - 基于业务维度、主体、客体匹配度筛选候选对
5. **LLM 判定与评审** - 双阶段验证，支持重试机制

- L4_opposite: 15% 权重（反向匹配 → 冲突）
- L3_weak: 0% 权重（弱匹配 → 剔除）

## 更新日志

### 2026-05-22 - 小批量测试验证

**变更内容**：
- 外规解析支持 JSON 格式 (`full_outer_data.json`，486MB，11,757部法规)
- 外规实体提取改用关键词匹配（内规语义要素作为词典），避免逐条调用LLM
- 实现令牌桶限流器 (`TokenBucketRateLimiter`)，替代单锁串行化
- 语义提取并行化 (`ThreadPoolExecutor` + 可配置并发数)
- 过滤索引缓存优化，避免重复构建倒排索引
- 硬筛选条件放宽：`biz_dim_match >= 1` + `subject_match >= 1`

**测试结果**：
- 200条候选对 → 148条训练数据
- **无关标签: 0%** （目标达成）
- 失败率: 0%

### 处理流程

1. **解析条款** (`node_parse_clauses`)
   - 读取外规/内规 DOCX 文件
   - 按条号切分条款
   - 调用 LLM 提取语义要素（subjects, objects, biz_dims, action）
   - 应用同义词归一化
   - 缓存语义数据到 `output/outer_clauses_with_semantics.json`

2. **分层筛选候选对** (`node_filter_candidates`)
   - 硬筛选：`biz_dim_match >= 1` 且 `subject_match >= 1`
   - 分层归类：
     - **L1_strong**: 客体匹配≥2 + action匹配，或主体匹配≥2 + 客体匹配≥1
     - **L2_moderate**: 客体匹配≥1 + action匹配，或客体匹配≥2
     - **L3_weak**: 不满足上述条件（剔除）
   - 按比例采样：L1 65%, L2 35%

3. **批量并发处理** (`node_batch_process`)
   - 20 并发 LLM 调用
   - 每个候选对：判断 → 评审 → (通过/重试/失败)
   - **新增**：对"完全符合"/"基本一致"的 pair 额外生成冲突场景 QA

4. **划分数据集** (`node_split_dataset`)
   - 条款级非重叠划分
   - 合并原始QA和冲突QA
   - 打乱顺序

5. **持久化存储** (`node_persist`)
   - 输出训练集/验证集 JSONL
   - 转换为 LLaMA Factory Alpaca 格式
   - 输出统计报告

### 1. 语义要素提取

从条款文本中提取四类语义要素：

| subjects | 法律主体 | 商业银行、借款人、监管机构 |
| objects | 法律客体 | 贷款、个人信息、不良资产 |
| biz_dims | 业务维度 | 信贷管理、反洗钱、公司治理 |
| action | 行为模态 | 应当、不得、可以、禁止 |

**Prompt 模板**：


### 3. 倒排索引优化

使用倒排索引加速候选对筛选，避免 O(n²) 遍历：

```python
def build_inverted_index(clauses: list[dict], field: str) -> dict[str, set]:
    """构建倒排索引"""
    index = defaultdict(set)
    for idx, clause in enumerate(clauses):
        values = clause.get(field, [])
        if isinstance(values, list):
            for v in values:
                if v:
                    index[v].add(idx)
    return index

# 示例：biz_dims 倒排索引
{
    "信贷管理": {0, 5, 12, 23, ...},
    "反洗钱": {1, 8, 15, ...},
    "公司治理": {2, 7, ...},
}
```

**筛选流程**：

```python
# 1. 构建内规倒排索引
inner_biz_dim_index = build_inverted_index(inner_clauses, "biz_dims")
inner_subject_index = build_inverted_index(inner_clauses, "subjects")

# 2. 对于每个外规条款，快速找到候选内规
for outer in outer_clauses:
    candidates = set()
    for biz in outer.get("biz_dims", []):
        candidates.update(inner_biz_dim_index.get(biz, set()))
    for subj in outer.get("subjects", []):
        candidates.update(inner_subject_index.get(subj, set()))
    # 只对候选内规计算匹配度
    for inner_idx in candidates:
        compute_similarity(outer, inner_clauses[inner_idx])
```

### 4. LLM 判定 Prompt

**五级分类标签**：

| 标签 | 定义 | 示例 |
|------|------|------|
| 完全符合 | 内规完全落实外规要求，语义要素完全对应 | 外规"不得向关系人发放信用贷款" → 内规"禁止向关系人发放信用贷款" |
| 基本一致 | 内规落实外规要求，但表述或范围有细微差异 | 外规"应当披露信息" → 内规"每季度披露财务报告" |
| 部分冲突 | 存在部分不一致，但核心要求仍有落实 | 外规"不得超过50%" → 内规"最高可达60%" |
| 直接冲突 | 与外规要求直接矛盾 | 外规"不得..." → 内规"可以..." |
| 无关 | 无对应关系 | 业务维度、主体、客体均不相关 |

**判定输出格式**：

```json

  "label": "完全符合",
  "confidence": 0.95,
  "reasoning": "内规完全落实外规要求，主体均为商业银行，客体均为贷款，行为模态均为禁止性规定",
  "key_evidence": ["主体一致", "客体一致", "行为模态一致"],
  "conflict_details": {
    "conflict_type": "无冲突",
    "description": "无"
  }
}
```

```python
    outer_ids = set(qa["outer_clause"].get("uid") for qa in qa_pairs)
    inner_ids = set(qa["inner_clause"].get("uid") for qa in qa_pairs)
    
    # 随机划分条款
    val_outer_ids = set(random.sample(outer_ids, int(len(outer_ids) * val_ratio)))
    val_inner_ids = set(random.sample(inner_ids, int(len(inner_ids) * val_ratio)))
    
    # 划分 QA 对

        outer_in_val = qa["outer_clause"].get("uid") in val_outer_ids
        inner_in_val = qa["inner_clause"].get("uid") in val_inner_ids
        
        if outer_in_val and inner_in_val:
            val_set.append(qa)
        else:
            train_set.append(qa)
```

## 核心创新点

### 1. 分层筛选策略

根据匹配特征将候选对分层，只保留高相关性的 L1+L2 层：

| 层级 | 条件 | 预期标签 |
|------|------|---------|
| L1_strong | obj>=2 + action=1, 或 subj>=2 + obj>=1 | 完全符合, 基本一致 |
| L2_moderate | obj>=1 + action=1, 或 obj>=2 | 部分冲突, 基本一致 |
| L3_weak | 其他（剔除） | 大概率无关 |

## 故障排查

### 候选对数量不足

**原因**：筛选条件太严格。

**解决方案**：
1. 放宽 L1/L2 层条件
2. 增大 `max_candidates_to_process`

### LLM 调用失败率高

**原因**：并发过高或网络问题。

**解决方案**：
1. 降低 `max_concurrent_calls`（推荐 10-15）
2. 检查 `llm.base_url` 和 `llm.api_key`

### "无关"比例过高

**原因**：筛选门槛太低。

**解决方案**：
1. 提高分层筛选条件（如 obj>=2）
2. 增大冲突生成比例
